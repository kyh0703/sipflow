<feature>
  <name>UAManager with goroutine-leak-free lifecycle</name>
  <files>internal/infra/sip/ua_manager.go, internal/infra/sip/ua_manager_test.go</files>
  <behavior>
    UAManager is a concurrent-safe registry mapping nodeID (string) to managed diago UA instances.

    UAConfig struct fields:
    - DisplayName string (SIP User-Agent header, e.g. "SIPFlow/1.0")
    - Transport string ("udp", "tcp", "tls")
    - BindHost string (default "0.0.0.0")
    - BindPort int (default 0 for ephemeral)

    UAStatus struct fields:
    - NodeID string
    - Transport string
    - Active bool

    Methods and behavior:
    - NewUAManager(logger *slog.Logger) *UAManager
    - CreateUA(nodeID string, cfg UAConfig) error
      - Creates sipgo.NewUA() with sipgo.WithUserAgent(cfg.DisplayName)
      - Creates diago.NewDiago(ua, diago.WithTransport(...), diago.WithLogger(...))
      - Stores in map with context.WithCancel for lifecycle
      - Returns error if nodeID already exists
      - Uses BindPort: 0 (ephemeral) to avoid port conflicts
    - DestroyUA(nodeID string) error
      - Cancels context, calls ua.Close()
      - Removes from map
      - Returns error if nodeID not found
    - DestroyAll()
      - Destroys all tracked UAs
      - Safe to call when empty
    - GetStatus(nodeID string) (UAStatus, bool)
      - Returns status of a specific UA
    - ListActive() []UAStatus
      - Returns list of all active UAs

    Test cases (TDD):
    - CreateUA("node-1", cfg) -> nil error, ListActive has 1 entry
    - CreateUA("node-1", cfg) twice -> error (duplicate)
    - DestroyUA("node-1") -> nil error, ListActive has 0 entries
    - DestroyUA("nonexistent") -> error
    - CreateUA + DestroyUA -> goleak.VerifyNone passes (NO goroutine leaks)
    - CreateUA x3 + DestroyAll -> goleak.VerifyNone passes
    - GetStatus("node-1") after create -> (status, true)
    - GetStatus("nonexistent") -> (zero, false)
  </behavior>
  <implementation>
    Use sync.RWMutex for concurrent access.
    managedUA struct: { dg *diago.Diago, ua *sipgo.UserAgent, cancel context.CancelFunc }
    DestroyUA: cancel context first, then ua.Close() to release transport.
    DestroyAll: collect IDs under read lock, then destroy each (avoid holding lock during Close).

    IMPORTANT API corrections from research:
    - Use sipgo.NewUA() NOT diago.NewUA() (doesn't exist)
    - Use diago.NewDiago(ua, opts...) to wrap the sipgo UA
    - Use diago.Transport{Transport: "udp", BindHost: "0.0.0.0", BindPort: 0}
    - ua.Close() releases transport listeners (not dg.Close which doesn't exist)

    For goleak tests: may need to ignore some background goroutines:
    - goleak.IgnoreAnyFunction("internal/poll.runtime_pollWait")
    - goleak.IgnoreTopFunction("time.Sleep") if needed
    Add a small sleep (50ms) after DestroyUA before goleak check to allow goroutine cleanup.
  </implementation>
</feature>

<feature>
  <name>SIP Trace Logger</name>
  <files>internal/infra/sip/sip_logger.go, internal/infra/sip/sip_logger_test.go</files>
  <behavior>
    SIPTraceHandler wraps slog.Handler and additionally calls a callback function with each log record.
    This decouples the trace capture from Wails event emission (callback is injected).

    - NewSIPTraceHandler(inner slog.Handler, callback func(SIPTraceEntry)) *SIPTraceHandler
    - SIPTraceEntry struct: Time (time.Time), Level (string), Message (string), NodeID (string, extracted from attrs)
    - Handle(ctx, record) -> calls inner.Handle AND calls callback with SIPTraceEntry
    - Enabled/WithAttrs/WithGroup delegate to inner handler

    Test cases:
    - Handle record -> callback receives SIPTraceEntry with correct fields
    - Handle record -> inner handler also receives record (passthrough)
    - Callback receives NodeID extracted from "node" attribute if present
  </behavior>
  <implementation>
    Standard slog.Handler wrapper pattern. Extract "node" attr by iterating record.Attrs.
    Keep it simple -- no buffering, no channels. Direct callback invocation.
    The Wails event emission callback will be injected when wiring in Plan 03.
  </implementation>
</feature>
