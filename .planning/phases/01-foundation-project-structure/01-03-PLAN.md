<objective>
Create Wails handler layer with FlowService bindings, structured error responses, and EventEmitter with handshake protocol that prevents race conditions.

Purpose: This is the Go-side communication layer. FlowService exposes CRUD operations to the frontend via Wails bindings. EventEmitter implements the backend side of the handshake protocol (wait for frontend:ready, then emit backend:ready) and provides safe event emission with 100us delay for rapid events.

Output: Wails-bindable services with structured responses and race-condition-safe event emission.
</objective>

  <name>Task 1: Structured response types and FlowService handler</name>
  <files>
    internal/handler/response.go
    internal/handler/flow_service.go
  </files>
  <action>
    1. Create `internal/handler/response.go`:
       - Define generic `Response[T any]` struct:
         ```go
         type Response[T any] struct {
             Success bool   `json:"success"`
             Data    T      `json:"data,omitempty"`
             Error   *Error `json:"error,omitempty"`
         }

         type Error struct {
             Code    string `json:"code"`
             Message string `json:"message"`
         }
         ```
       - Constructor functions:
         - `Success[T any](data T) Response[T]` - returns success response
         - `Failure[T any](code, message string) Response[T]` - returns error response

    2. Create `internal/handler/flow_service.go`:
       - FlowService struct with `entClient *ent.Client` field
       - Constructor: `NewFlowService(client *ent.Client) *FlowService`
       - Methods (all return `Response[T]` types, all use context.Background()):

         `CreateFlow(name, description string) Response[*ent.Flow]`:
         - Validates name is not empty (return Failure with "VALIDATION_ERROR" if empty)
         - Creates flow via `entClient.Flow.Create()`
         - Returns Success with created flow

         `GetFlow(id int) Response[*ent.Flow]`:
         - Queries flow by ID with eager loading of nodes and edges
         - Returns Failure with "NOT_FOUND" if not found
         - Returns Success with flow

         `ListFlows() Response[[]*ent.Flow]`:
         - Queries all flows ordered by updated_at desc
         - Returns Success with flow list

         `DeleteFlow(id int) Response[bool]`:
         - Deletes flow by ID (cascade should handle nodes/edges via ent)
         - Returns Failure with "NOT_FOUND" if not found
         - Returns Success(true)

       Important:
       - These methods are PUBLIC (capitalized) because Wails binding requires exported methods
       - Use ent's generated query builders, not raw SQL
       - Keep methods thin - they translate between Wails/HTTP concerns and ent. Business logic will move to usecase layer when needed (not in Phase 1)
       - For Phase 1, FlowService directly uses ent client (no usecase layer yet - YAGNI). The usecase layer will be introduced when business logic complexity warrants it.
  </action>

    - `go vet ./internal/handler/...` has no warnings
    - FlowService has 4 public methods (CreateFlow, GetFlow, ListFlows, DeleteFlow)
    - Response types use generics correctly
  </verify>

<task type="auto">
  <name>Task 2: EventEmitter with handshake protocol and Wails binding registration</name>
  <files>
    internal/handler/event_emitter.go
    app.go
    main.go
  </files>
  <action>
    1. Create `internal/handler/event_emitter.go`:
       - EventEmitter struct with fields:
         - `ctx context.Context` (Wails app context, set during startup)
         - `frontendReady bool` (tracks handshake state)
         - `readyChan chan struct{}` (signals when frontend is ready)

       - Constructor: `NewEventEmitter() *EventEmitter` - initializes readyChan

       - `SetContext(ctx context.Context)`: Called during app startup to inject Wails context

       - `OnStartup(ctx context.Context)`: Called by Wails lifecycle. Sets context, registers EventsOn listener for "frontend:ready" that:
         a. Sets `frontendReady = true`
         b. Emits "backend:ready" event
         c. Closes readyChan to signal any waiting goroutines

       - `WaitForFrontend(timeout time.Duration) error`: Blocks until frontend signals ready or timeout. Uses select on readyChan and time.After.

       - `Emit(eventName string, data ...interface{})`: Emits event via `runtime.EventsEmit`. Only emits if frontendReady is true. Logs warning if frontend not ready.

       - `EmitSafe(eventName string, data ...interface{})`: Same as Emit but adds `time.Sleep(100 * time.Microsecond)` after emission to prevent rapid event race conditions (from research: Wails issue #2759).

    2. Update `app.go`:
       - Add `eventEmitter *handler.EventEmitter` field to App struct
       - Add `flowService *handler.FlowService` field to App struct
       - In constructor (NewApp or equivalent), create EventEmitter and FlowService instances
       - In `startup(ctx context.Context)` method:
         a. Initialize ent client (from Plan 01)
         b. Run auto-migration
         c. Create FlowService with ent client
         d. Set EventEmitter context
         e. Call EventEmitter.OnStartup(ctx)
       - In `beforeClose(ctx context.Context)` or `shutdown`:
         a. Close ent client

    3. Update `main.go`:
       - In `wails.Run()` options, add `Bind` array with:
         - `app.flowService`
         - `app.eventEmitter` (if it needs direct frontend calls)
       - Or bind FlowService and EventEmitter as separate bound structs
       - The Bind array tells Wails which Go structs to generate TypeScript bindings for

    Important:
    - Wails generates TypeScript bindings from bound structs. Only PUBLIC methods on bound structs become callable from frontend.
    - EventEmitter.OnStartup uses `runtime.EventsOn` from `github.com/wailsapp/wails/v2/pkg/runtime`
    - The readyChan pattern is thread-safe for the handshake
    - Do NOT use mutex for frontendReady flag - the chan-based pattern is cleaner and idiomatic Go
  </action>

    - In browser console (during wails dev), `window.go.handler.FlowService.ListFlows()` should return a response
  </verify>
