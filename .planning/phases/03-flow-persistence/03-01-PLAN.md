<objective>
Establish project-based .sipflow file infrastructure with runtime SQLite database switching and Wails native menu.

Purpose: Phase 3 transitions from a single app-internal database to per-project .sipflow files. Each project is an independent SQLite database that users can share, copy, and version control. This plan creates the Go backend infrastructure for opening/closing project files and the Wails File menu.

Output: ProjectService with runtime DB switching, updated ent schema with viewport fields, Wails native File menu (New/Open/Save As).
</objective>

  <action>
1. Update `ent/schema/flow.go` to add viewport fields for canvas state persistence:
   - `viewport_x` float64, default 0
   - `viewport_y` float64, default 0
   - `viewport_zoom` float64, default 1
   These store the xyflow viewport so users see exact canvas position on reload.

2. Run `cd frontend && npm run generate` or the ent generate command (`go generate ./ent`) to regenerate ent code after schema change.

3. Create `internal/handler/project_service.go` with ProjectService struct:
   ```go
   type ProjectService struct {
       ctx           context.Context
       entClient     *ent.Client
       currentPath   string
       flowService   *FlowService
   }
   ```

   Methods:
   - `NewProjectService(flowService *FlowService) *ProjectService`
   - `SetContext(ctx context.Context)` - called from app.startup
   - `NewProject() Response[bool]` - Show Wails SaveFileDialog with .sipflow filter, create new DB file at chosen path, run schema migration, close previous client if any, set currentPath. Return error if user cancels dialog.
   - `OpenProject() Response[string]` - Show Wails OpenFileDialog with .sipflow filter, open DB file, run schema migration (Schema.Create - idempotent), close previous client if any, update FlowService's entClient via SetEntClient, set currentPath. Return the path on success.
   - `CloseProject() Response[bool]` - Close current ent client, clear currentPath, set FlowService entClient to nil.
   - `SaveProjectAs() Response[string]` - Show SaveFileDialog, copy current DB file to new path (use io.Copy), reopen at new path, return new path. If no project open, return error.
   - `CurrentProjectPath() string` - Return currentPath (empty string if no project open).
   - `IsProjectOpen() bool` - Return currentPath != "".
   - `openDatabase(path string) error` - Private helper: close existing client, call sqlite.OpenEntDriver(path), create ent.NewClient, run Schema.Create, update flowService.SetEntClient. This is the core DB switching logic used by NewProject, OpenProject, SaveProjectAs.

   Use Wails runtime for dialogs:
   ```go
   import "github.com/wailsapp/wails/v2/pkg/runtime"

   path, err := runtime.SaveFileDialog(s.ctx, runtime.SaveDialogOptions{
       Title: "Create New Project",
       DefaultFilename: "untitled.sipflow",
       Filters: []runtime.FileFilter{
           {DisplayName: "SIPFlow Project", Pattern: "*.sipflow"},
       },
   })
   ```

   IMPORTANT: Always close previous ent client before opening new one to prevent "database is locked" errors. Use a mutex or sequential access pattern since Wails calls are already single-threaded from frontend.

4. Update `internal/infra/sqlite/driver.go` - no changes needed, existing OpenEntDriver works for any path. Just verify it handles paths with spaces correctly (it does, since it uses fmt.Sprintf with file: prefix).

  <action>
1. Update `app.go`:
   - Add `projectService *handler.ProjectService` to App struct.
   - In NewApp(), create ProjectService: `projectService: handler.NewProjectService(flowService)` where flowService is created first.
   - In startup(), call `a.projectService.SetContext(ctx)` instead of the current single-DB initialization. REMOVE the existing database initialization code (configDir, dbPath, sqlite.OpenEntDriver, ent.NewClient, Schema.Create, flowService.SetEntClient). The app now starts with NO database open.
   - Keep eventEmitter setup as-is.
   - In shutdown(), close project via `a.projectService.CloseProject()` instead of directly closing entClient. Remove direct entClient reference from App struct since ProjectService owns it now.

2. Update `main.go`:
   - Add Wails native menu using `menu.NewMenu()`:
     ```go
     import "github.com/wailsapp/wails/v2/pkg/menu"
     import "github.com/wailsapp/wails/v2/pkg/menu/keys"

     appMenu := menu.NewMenu()
     fileMenu := appMenu.AddSubmenu("File")
     fileMenu.AddText("New Project", keys.CmdOrCtrl("n"), func(cd *menu.CallbackData) {
         app.projectService.NewProject()
     })
     fileMenu.AddText("Open Project...", keys.CmdOrCtrl("o"), func(cd *menu.CallbackData) {
         app.projectService.OpenProject()
     })
     fileMenu.AddSeparator()
     fileMenu.AddText("Save", keys.CmdOrCtrl("s"), func(cd *menu.CallbackData) {
         // Save is handled by frontend via Wails binding (needs flow state from frontend)
         // Emit event to frontend to trigger save
         wailsRuntime.EventsEmit(app.ctx, "menu:save")
     })
     fileMenu.AddText("Save As...", keys.CmdOrCtrl("shift+s"), func(cd *menu.CallbackData) {
         app.projectService.SaveProjectAs()
     })
     ```
   - Add `Menu: appMenu` to wails.Run options.
   - Add `app.projectService` to the Bind slice.

   IMPORTANT: The "Save" menu item emits an event to frontend because saving requires the current xyflow canvas state (nodes, edges, viewport) which lives in the frontend. The frontend listens for "menu:save" and calls the backend SaveFlow method with the current state. New/Open/SaveAs are pure backend operations that don't need frontend state.

3. Emit events to notify frontend of project state changes:
   - After successful OpenProject: `runtime.EventsEmit(s.ctx, "project:opened", path)`
   - After successful NewProject: `runtime.EventsEmit(s.ctx, "project:created", path)`
   - After CloseProject: `runtime.EventsEmit(s.ctx, "project:closed")`
   These events let the frontend react (clear canvas, load flow list, update title bar).
